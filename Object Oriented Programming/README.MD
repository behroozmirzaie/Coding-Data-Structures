۲۵- مفهوم destructor را توضیح دهید.

یک Destructor متد خاص برای یک کلاس است و هنگامی که یک شی در نهایت از بین می رود به طور خودکار فراخوانی می شود. نام
destructor نیز با نام کلاس یکی است اما با پیشوند tilde (~) دنبال می شود.

یک destructor برای آزاد کردن حافظه تخصیص یافته به صورت پویا و آزاد کردن منابع استفاده می شود. با این حال، می‌توانید یک
روش سفارشی را پیاده‌سازی کنید که به شما امکان می‌دهد با فراخوانی destructor، تخریب شی را کنترل کنید.

مثال پایتون:

```python
class Employee:
 
    # Initializing
    def __init__(self):
        print('Employee created.')
 
    # Deleting (Calling destructor)
    def __del__(self):
        print('Destructor called, Employee deleted.')
 
obj = Employee()
del obj
```

```python

# Python program to illustrate destructor
 
class A:
    def __init__(self, bb):
        self.b = bb
 
class B:
    def __init__(self):
        self.a = A(self)
    def __del__(self):
        print("die")
 
def fun():
    b = B()
 
fun()
```

ماژول های OOP به یکدیگر وابسته هستند. Coupling به سطح وابستگی بین دو ماژول نرم افزاری اشاره دارد. دو ماژول را به یکدیگر
وابسطه میگیم زمانی که در یک ماژول تغییری بدهیم و برای اینکه ماژول دوم تغییرات را پشتیبانی کند باید ماژول دوم را هم تغییر
دهیم.

تکنیک هایی مثل Inversion of Control و Dependency injection برای بدست آوردن loose coupling استفاده می شود.

۲۷- دقیقاً تفاوت بین یک interface و یک کلاس abstract چیست؟

یک interface یک قرارداد است: شخصی که interface را می‌نویسد می‌گوید، "هی، من می‌پذیرم همه چیز به این شکل باشد" و شخصی که
از interface استفاده می‌کند می‌گوید "OK، کلاسی که من می‌نویسم به همان شکل است". یک interface یک پوسته خالی است. فقط
امضای method ها در اون وجود دارند که نشان می دهد method ها بدنه ندارند. interface نمی تواند هیچ کاری انجام دهدو فقط یک
الگو است. پیاده‌سازی یک interface میزان CPU بسیار کمی مصرف می‌کند، زیرا یک کلاس نیست، فقط یک سری نام است، و بنابراین
جستجوی گران قیمتی برای انجام دادن آن وجود ندارد.

کلاس های Abstract، بر خلاف رابط ها، یم کلاس هستند. استفاده از آن‌ها گران‌تر است، زیرا زمانی که از آن‌ها به ارث می‌برید،
باید جستجویی انجام شود. کلاس های Abstract بسیار شبیه interface ها هستند، اما چیزهای بیشتری دارند: می توانید یک رفتار
برای آنها تعریف کنید. بیشتر درباره این است که فردی می گوید: "این کلاس ها باید اینطور به نظر برسند، و آنها در این موارد
مشترک هستند، پس جاهای خالی را پر کنید!".

۲۸- انواع وراثت را توضیح دهید.

در شی گرایی وراثت چهار مدل دارد.

الف) Single inheritance - شامل یک کلاس پایه و یک کلاس مشتق شده است

ب) Hierarchical inheritance - شامل یک کلاس پایه و چندین کلاس مشتق شده از یک کلاس پایه است

ج) Multilevel inheritance - شامل یک کلاس مشتق شده از یک کلاس مشتق شده است

د) Multiple inheritance - شامل چندین کلاس پایه و یک کلاس مشتق شده است

۲۹- منظور از Cohesion در برنامه نویسی شی گرا چیست؟

در OOP ما کد خود را در ماژول ها توسعه می دهیم. هر ماژول دارای مسئولیت های خاصی است. Cohesion نشان می دهد که چقدر مسئولیت
های ماژول با یکدیگر مرتبط هستند. انسجام بالاتر همیشه ترجیح داده می شود. به صورت ساده "انسجام(Cohesion) بر پیچیدگی درون
یک ماژول تمرکز دارد."

۳۰- چگونه Circular Reference را حل می کنید؟

برای مسئله ای به این شکل که کلاس A، کلاس B را به عنوان یک properties دارد، و همینطور کلاس B، کلاس A را به صورت
properties دارد. این مشکل را چگونه حل می کنید؟

اول از همه اینکه باید توی طراحی که انجام دادید تجدید نظر کنید. زمانی که مشکل circular reference داریم مشخص می شود که در
طراحی نقص هایی وجود داشته است.

در بیشتر موارد زمانی که مجبوریم دو کلاس به یکدگیر ارجاع داده شوند از یک interface استفاده می کنیم. به عنوان مثال:

##### Before

```Java
public class Foo
{
    Bar myBar;
}

public class Bar
{
    Foo myFoo;
}
```

##### Dependency graph:

```Java
Foo     Bar
 ^       ^
 |       |
Bar     Foo
```

همانطور که مشاهده میکنید Foo به Bar وابسطه است و هیمنطور Bar به Foo وابسطه است.

##### AFTER

```Java
public interface IBar
{
}

public class Foo
{
    IBar myBar;
}

public class Bar : IBar
{
    Foo myFoo;
}

```

##### Dependency graph:

```Java
Foo, IBar     IBar
    ^          ^
    |          |
   Bar        Foo

```

هر دوی Foo و Bar الان به Ibar وابسطه هستند. دیگه چیزی به اسم circular reference نداریم.

internal اعلام کرد.

۳۲- تفاوت بین association، aggregation و composition چیست؟

الف) Association رابطه ای است که در آن همه اشیا چرخه حیات خود را دارند و مالکی وجود ندارد.

بیایید یک مثال از معلم و دانشجو را در نظر بگیریم. چند دانش آموز می توانند با یک معلم و یک دانش آموز می تواند با چندین
معلم معاشرت(associate) کنند، اما هیچ مالکیتی بین اشیا وجود ندارد و هر دو چرخه حیات خود را دارند. هر دو را می توان به طور
مستقل ایجاد و حذف کرد.

ب) Aggregation یک شکل تخصصی از Association است که در آن همه اشیا چرخه حیات خود را دارند.اما مالکیت وجود دارد و اشیاء
فرزند نمی توانند به شی والد دیگری تعلق داشته باشند.

بیایید یک نمونه از دپارتمان و معلم را در نظر بگیریم.(Department and teacher). یک معلم نمی تواند به چندین دپارتمان تعلق
داشته باشد،اما اگر دپارتمان را حذف کنیم، شیء معلم از بین نمی رود. ما می توانیم در مورد آن به عنوان یک رابطه "has-a" فکر
کنیم.

ج) Composition دوباره یک شکل تخصصی از Association است و می‌توانیم آن را رابطه «مرگ» بنامیم(death relationship). یک نوع
قوی از Aggregation است. اشیاء Child چرخه حیات خود را ندارند و اگر شی والد حذف شود، تمام اشیاء فرزند نیز حذف خواهند شد.

بیایید دوباره مثالی از رابطه House و Rooms بزنیم. خانه می تواند چندین اتاق داشته باشد - زندگی مستقلی برای اتاق وجود
ندارد و هر اتاقی نمی تواند به دو خانه متفاوت تعلق داشته باشد. اگر خانه را حذف کنیم - اتاق به طور خودکار حذف می شود.
